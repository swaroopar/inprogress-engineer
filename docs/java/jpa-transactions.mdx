# JPA Transactions

In Spring, you add the **@Transactional** annotation on a method.
The framework then starts a session when the method begins.
It commits all transactions in the thread when the method returns.

A flow may have many @Transactional methods.
They share the same transaction, the outermost one, unless you say otherwise.

:::tip methods without **@Transactional**
For methods without this annotation:

1. Calls through spring-data CrudRepository already have this annotation.
   The transaction is handled there.
2. With direct calls, each query is its own transaction.
   It runs and commits on its own.
   :::

## Thread Safe Components

The following components are thread-safe in a JPA transaction

1. Entity Manager - All work with the database and the persistent context goes through this.
2. Persistent Context - Think of it as a map of DB entities the thread knows.
   It's also called the L1 cache.

## Shared Components

The following components are shared across threads

1. L2 Cache - It can share DB entities across many threads.
   For example, Hibernate clears a cache entry when an entity is updated.
2. Connection Pool - The database connections are created once and
   shared between threads.

:::important connection pools
The connection pool keeps the DB connection alive at all times.

It just connects to the database URL, sends the credentials,
and stays ready to start a session.

It uses TCP keep-alive to keep the connection active
while the pool is active.
:::

## Proxy Classes

The Spring framework doesn't weave the start and end code into the real classes.
Instead, if a class has a **transactional** annotation,
Spring creates a proxy class and its bean. The calling app then uses these.

This proxy bean first starts the transaction. Then it calls the real bean.

:::danger calling annotated classes directly
If you create and call a class directly, the proxy is skipped. The transaction isn't managed.
:::
